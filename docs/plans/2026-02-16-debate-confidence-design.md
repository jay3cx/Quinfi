# 多空辩论置信度重构设计（决策可靠优先）

- 日期: 2026-02-16
- 主题: 将“模型自评置信度”升级为“系统可校准置信度”
- 目标: 同时服务用户展示与系统决策，且优先保障决策可靠性
- 决策策略: 先复核，不达标再降级为观望

## 1. 背景与问题

当前多空辩论流程中，Bull/Bear/Judge 阶段的 `confidence` 由大模型直接输出。该做法有三个问题：

1. 可重复性不足：同类输入可能给出离散分值，难以稳定驱动自动门控。
2. 可解释性不足：分值来源不透明，无法明确为何触发或未触发风险动作。
3. 可校准性不足：缺少与真实后验表现的映射，难以形成长期可靠的阈值体系。

## 2. 设计目标

1. 保留辩论内容质量，但不再将模型自评分作为唯一决策依据。
2. 建立可解释、可门控、可逐步校准的系统置信度。
3. 支持三态决策：`pass` / `review` / `degrade`。
4. 满足“先复核，再降级”业务策略。
5. 在无历史标签条件下可立即上线，后续可平滑演进到监督校准。

## 3. 方案对比与选型

### 方案 A: 纯规则置信度

- 核心: 使用证据完备度、结构完整度、异常命中等规则打分。
- 优点: 快速、可控、可解释。
- 缺点: 与真实收益结果相关性初期有限。

### 方案 B: 纯一致性置信度

- 核心: 重跑辩论（多次或多模型）并用一致性衡量可靠性。
- 优点: 不依赖历史标签。
- 缺点: 成本与时延较高，缺少规则兜底会有盲区。

### 方案 C: 两层置信度（选定）

- 在线层: 规则分 + 一致性分，实时门控。
- 校准层: 持续沉淀样本，后续做监督校准。
- 结论: 当前阶段采用方案 C，兼顾即用性与长期准确性。

## 4. 架构设计

在现有辩论链路后增加 `Confidence Engine`，并将模型自评分降为辅助信号。

1. Debate Core（保留）
- 维持现有 DataGather/Bull/Bear/Rebuttal/Judge 生成流程。

2. Online Scoring（新增）
- 产出：`evidence_score`、`integrity_score`、`consistency_score`、`self_score(low-weight)`。

3. Decision Gate（新增）
- 输出：`pass` / `review` / `degrade`。
- 策略：先复核，不达标降级观望。

4. Calibration Layer（预留）
- 先做样本沉淀与统计分析。
- 后续训练校准器映射到真实可靠概率。

5. Observability（增强）
- 全量记录评分细节、命中规则、门控路径，支持回放和阈值调优。

## 5. 数据流与阈值策略

### 5.1 S0 初判分（无需历史标签）

信号定义：

- `evidence_score`: 论据可核验度（数值/时间/对象/来源映射）。
- `integrity_score`: 结构完整度（字段完整、阶段成功率、解析稳定性）。
- `self_score`: LLM 自评分（仅低权重参考）。

组合公式：

`S0 = 0.5 * evidence_score + 0.4 * integrity_score + 0.1 * self_score`

### 5.2 复核触发条件

触发 `review` 当满足任一条件：

- `S0 < 75`
- 命中硬规则（阶段失败、关键数据缺失、解析降级等）

复核动作：

- 发起一次二次辩论（低温度或异构模型）
- 计算 `consistency_score`（结论一致性、关键论据重合度、风险项一致性）

### 5.3 S1 终判分与门控

`S1 = 0.7 * S0 + 0.3 * consistency_score`

门控规则：

- `pass`: `S1 >= 70` 且未命中硬规则
- `degrade`: 其他情况全部降级为“观望/证据不足”

说明：复核最多一次，不允许 `review -> review` 循环。

## 6. 硬规则（高优先级）

以下条件命中时直接降级或封顶：

1. 核心阶段失败（Bull/Bear/Judge 任一失败）
2. JSON 解析失败且进入 fallback 文本
3. 关键数据源缺失（基金信息或净值）
4. 双方论点异常同向且无充分解释

推荐动作：

- 命中硬规则时置信度上限封顶（如 `max 60`）
- 严重缺失直接 `degrade`

## 7. 组件划分（最小改动）

1. `internal/debate/confidence_engine.go`（新增）
- 输入：`DebateResult` + 运行元信息
- 输出：`ConfidenceAssessment`
  - `s0` / `consistency_score` / `s1`
  - `decision` (`pass|review|degrade`)
  - `reasons[]`

2. `internal/debate/reviewer.go`（新增）
- 负责一次复核重跑
- 输出结构化复核结果用于一致性计算

3. `internal/debate/orchestrator.go`（改造）
- 六阶段后执行 `ConfidenceEngine.Evaluate(...)`
- 若 `review`，触发 reviewer，再做 S1 判定
- 若 `degrade`，强制建议语义降级为观望

4. `internal/debate/model.go`（扩展字段）
- `system_confidence`
- `decision_gate`
- `confidence_reasons[]`
- `review_attempted`

5. 日志与观测（增强）
- 结构化日志新增：`s0`、`s1`、`decision`、`hard_rule_hits`、`review_latency_ms`

## 8. 错误处理策略

1. 复核超时/失败
- 不阻塞主流程，按惩罚后的 `S1` 直接走 `degrade`

2. 解析失败
- 打硬规则并封顶，避免高分误判

3. 关键数据缺失
- 直接 `degrade` 并在 `reasons[]` 写明缺失项

4. 循环保护
- 最多复核 1 次

5. 配置保护
- 阈值配置读取失败回退默认值并告警

## 9. 测试与验收标准

### 9.1 单元测试

- S0/S1 计算正确性
- 边界值行为（69/70/75）
- 硬规则优先级

### 9.2 集成测试

- `pass` 路径
- `review -> pass` 路径
- `review -> degrade` 路径
- `review` 失败降级路径

### 9.3 回归测试

- 现有辩论结果结构兼容
- 新字段序列化稳定
- UI 不因新增字段退化

### 9.4 验收门槛

1. 新增测试通过
2. 影子模式观察期内降级案例可解释
3. 解析失败或关键数据缺失场景不得 `pass`

## 10. 分阶段落地建议

1. 第 1 阶段（立即）
- 上线在线层（S0 + 一次复核 + 门控）
- 不改模型内容生成逻辑，只改决策链路

2. 第 2 阶段（数据沉淀）
- 增加样本归档：输入特征、门控结果、后续表现标签

3. 第 3 阶段（监督校准）
- 周期性训练校准器（isotonic/logistic）
- 用校准概率替代当前固定阈值部分逻辑

## 11. 非目标

1. 本次不做收益预测模型
2. 本次不做多轮复核策略优化
3. 本次不做跨资产统一置信度体系

## 12. 风险与缓解

1. 复核增加延迟
- 缓解：仅在 `review` 触发时执行，保留超时降级

2. 初期阈值不稳
- 缓解：影子模式 + 结构化观测 + 小步调参

3. 规则过严导致过多降级
- 缓解：引入原因分解，区分数据问题与模型不稳定问题

## 13. 预期收益

1. 决策链路从“主观分”转为“系统门控分”
2. 低质量样本更易被拦截，误导性确定性建议下降
3. 为后续监督校准和长期可靠性打下数据基础
